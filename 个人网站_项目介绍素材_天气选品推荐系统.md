# 个人网站用：天气驱动服装选品推荐系统（素材稿）

> **用途说明**：本文档供日后搭建个人作品集网站时，由 AI 或人工摘取段落、改写语气、配图与排版。前半部分偏 **企业价值**（解决什么问题、优势是什么）；后半部分保留功能与技术索引。避免夸大销量效果；技术细节以仓库内代码为准。

## 一、企业面临什么问题？本系统解决什么？

服装、鞋服、直播电商等企业，在「要不要推厚外套、要不要加连衣裙库存、直播排品怎么排」这类问题上，经常出现 **外部信号（天气）** 和 **内部事实（库存与经营数据）** 各说各话：

| 企业常见痛点 | 本系统带来的帮助 |
|-------------|------------------|
| **天气变了，但不知道对应该推哪类货** | 把未来一段时间的气温、降水、湿度、风力等，**自动映射**到企业 BI 里真实存在的品类/款式空间，减少凭感觉选大类。 |
| **运营看天气，买手看报表，信息不在一起** | 把 **天气预报** 与 **观远 BI 中的商品明细**（如库存、在途、转化、退款等，以企业实际字段为准）拉通，在同一套结果里呈现。 |
| **推荐很「准」，但仓库里没货或货很差** | 在命中「天气适合的品类」之后，再用 **库存与经营指标加权排序**，优先露出**能卖、值得推**的 SKU，降低「推荐了但卖不动/没现货」的尴尬。 |
| **只看本地城市，忽略全国大盘** | 支持结合 **重点城市/区域** 的天气信号（如多雨、高湿）做叠加判断，更贴近直播与全国电商的成交结构（实现以系统配置为准）。 |
| **规则写在 Excel 里，改一次要 IT 排期** | 选品规则可 **配置化**（如独立 JSON），业务侧可迭代温度带、品类白名单、雨天叠加策略，而不必每次改核心代码。 |
| **BI 接口不稳或改版，大屏就挂** | 具备 **缓存、超时控制、最近一次成功快照回退** 等机制，在数据暂时不可用时仍能给出上一版可用结果，适合内部长期运行。 |
| **结论散落在群聊，难沉淀** | 提供 **Web 仪表板** 统一展示；可选把结果 **同步到飞书多维表**，方便开会对齐、留痕与协作。 |

**概括成一句给老板听的话**：  
在「天气变化」和「公司此刻能卖什么货」之间搭一座自动桥，让运营、直播、买手在 **同一事实与同一套规则** 下做选品与备货讨论，而不是靠碎片信息与个人经验。

## 二、核心优势（对外介绍时可当「为什么选择这套方案」）

1. **决策链闭环，而不是两个孤立工具**  
   不是单独做一个「天气小程序」再做一个「BI 报表」；而是 **天气信号 → 品类映射 → 结合真实库存与经营指标的排序 → 可执行清单**，直接服务选品与排品动作。

2. **与企业现有 BI 投资衔接**  
   企业往往在观远等 BI 里已沉淀了最可信的库存与销售明细；本系统通过 **标准 Public API 方式** 取数，减少重复建数仓、重复维护 Excel 的需求。

3. **可解释、可调参**  
   推荐依据来自 **可配置规则 + 明确权重思路**（如侧重可售库存，兼顾转化与退款），便于业务方理解「为什么今天推这类」，也便于按季节与大促逐步调参。

4. **面向真实运维环境**  
   考虑 Token、网络超时、BI 视图差异、页面多 Tab 导致卡片难定位等 **落地问题**，并配有快照回退，偏向「能长期跑」的内部系统，而非演示 Demo。

5. **协作与部署友好**  
   密钥与环境与代码分离；结果可进飞书多维表；支持容器/云平台部署，适合企业内网或受控环境对外提供服务。

6. **扩展空间清晰**  
   增城市、增品牌维度、增数据源卡片、接更多协作工具，都可以在现有「配置 + 服务模块」思路上延伸，不必推倒重来。

## 三、一句话与定位

- **项目名称（工作名）**：天气驱动的服装选品推荐系统 V2（`weather_v2`）
- **一句话**：将 **多城市天气预报** 与 **企业 BI 实时商品明细** 打通，用可配置的规则与评分模型，输出 **品类级 + SKU 级** 的选品与补货参考，并通过 Web 仪表板呈现；可选同步至 **飞书多维表**。
- **类型**：面向企业的 **运营决策支持** 类 Web 应用（Flask API + 仪表板），强调「可执行推荐」而非仅数据展示。

## 四、核心功能（对外可写进「功能列表」）

| 能力 | 说明 |
|------|------|
| 多城市天气 | 基于 **Open-Meteo** 拉取预报（无需 API Key），支持多城并行；服务端带缓存，降低请求频率。 |
| 天气 → 品类映射 | 将均温、降水、湿度、风力等信号映射到 **BI 侧品类/款式空间**；规则可来自业务 Excel 提炼，并支持 **`recommendation_rules.json` 等外部规则文件** 调整。 |
| BI 库存与经营指标 | 通过 **观远（Guandata）Public API** 从指定页面/卡片拉取明细（如库存、在途、退款率、转化率等字段，以实际卡片为准）；支持 **多页面/多卡片配置**（`guandata_sources.json` + 环境变量）。 |
| 韧性设计 | API 异常或字段不匹配时，可回退到 **本地最近一次成功快照**，避免仪表板完全空白。 |
| 智能推荐与排序 | 在天气命中品类后，结合库存与经营指标做 **加权评分**（例如侧重可售库存，辅以转化与退款），输出 Top 品类与 Top 单品。 |
| 全国/区域信号 | 支持将「重点城市多雨、高湿、大风」等作为 **叠加信号**，与本地城市规则组合（实现以代码为准）。 |
| 多品牌/前缀 | 支持按品牌维度过滤或识别（配置与别名映射，便于多店铺或子品牌）。 |
| Web 仪表板 | 响应式页面：天气概览、推荐品类与单品卡片、采购/运营向提示。 |
| 简单权限 | 提供基础登录与角色（便于内网或演示环境控制访问）。 |
| 飞书集成 | 可将推荐结果 **同步到飞书多维表**（需配置 App 与多维表 Token，密钥走环境变量）。 |
| 运维友好 | 提供健康检查、数据刷新接口；支持 Docker / 自建服务器部署（见 `Dockerfile`、`scripts/deploy-weather.sh`）。 |

## 五、技术架构（给「技术栈/架构图」段落用）

- **后端**：Python 3 + **Flask**（REST JSON API、`flask-cors`）
- **天气**：**Open-Meteo** HTTP API
- **BI**：自研 **`guandata.py`** 连接器（登录、页面/卡片枚举、卡片取数、Token 缓存）；库存业务封装在 **`bi_inventory_service_v3.py`**
- **推荐与规则**：**`bi_data_service.py`**（映射、评分、与城市/全国信号的组合逻辑）+ **`recommendation_rules.json`**
- **前端**：**Jinja2 模板** + 静态资源（仪表板型单页体验）
- **配置**：**python-dotenv**（`.env` 本地、平台环境变量生产）；**`guandata_sources.json`** 管理多数据源
- **协作出口**：**飞书开放平台** 多维表写入（`feishu_service.py`）

数据流（文字版，方便画箭头图）：

`Open-Meteo → 天气服务 → 推荐引擎 ← 观远 BI（Public API）← 配置（页面/卡片/关键字）`  
`→ JSON API → 前端仪表板`  
`→（可选）飞书多维表`

## 八、事实边界与诚实表述（建站时建议保留）

- 推荐结果是 **决策辅助**，不是销量承诺；效果依赖 BI 字段质量、规则设计与业务校准。  
- 天气 API 与 BI API 均有 **可用性与速率限制**，系统通过缓存与快照缓解，但不能保证实时性达到秒级。  
- 观远侧 **页面结构若大幅改版**，可能需要调整卡片定位方式或字段校验逻辑。  
- 仓库内 README 部分历史描述可能与当前实现不一致，**以 `bi_inventory_service_v3.py`、`guandata.py` 与配置文件为准**。
