# 取数与推荐逻辑基线

## 1. 商品取数逻辑（当前正确版本）

- 数据源：观远 BI 卡片（`bi_inventory_service_v3.py`）
- 主路径：优先走 API 拉取真实文本数据；API 无数据时再走 Playwright 兜底抓取
- 字段口径：
  - `stock` = 款主仓实际库存
  - `sales_return_stock` = 款销退仓库存
  - `inbound` = 款采购在途数
  - `sales_return_inbound` = 款销退在途数
  - `virtual_cloud_stock` = 虚拟云仓库存（无字段时按 0）
  - `order_occupied` = 订单占有数
  - `available_stock` = (`stock` + `sales_return_stock` + `inbound` + `sales_return_inbound`) - `virtual_cloud_stock` - `order_occupied`
  - `return_rate`、`conversion_rate` 统一转为 0~1 小数
- 名称口径：
  - 优先小名（`name`）
  - 小名异常时回退到产品名称（`product_name`）
  - 再异常时回退到款式编号
- 失败兜底：
  - 使用最近一次成功抓取快照 `bi_inventory_last_success.json`
  - 不用模拟数据污染真实结果（除非显式 `use_bi_data=False`）

## 2. 智能采购建议逻辑（本次修订）

- 品类推荐：由天气规则命中得到推荐品类集合
- 品类结构分析：对“推荐品类下的全部商品”做聚合，不做在库门槛过滤
- 推荐单品：仅在单品层应用门槛
  - 条件：`stock > 200 and inbound == 0`
  - 高湿场景下，额外约束速干候选优先

## 3. 天气取数逻辑

- 数据源：Open-Meteo 7 天预报（北京/上海/广州）
- 取数字段：天气现象、最高/最低温、降水概率、湿度、风向风速、UV、日出日落
- 失败兜底：
  - 使用 mock 数据仅用于服务可用性兜底
  - mock 温度已改为“非单调波动”，避免出现“每天递增”误导

## 4. 天气推荐商品框架（讨论稿）

### 输入层（天气与业务上下文）

- 城市维度：`city`
- 天气核心：`avg_temp`、`rain_days`、`temp_trend`
- 体感修正：`city_temp_offset`（城市温差）
- 环境修正：`avg_humidity`、`avg_wind_scale`
- 区域修正：`national_rainy_city_count`
- 时令修正：`season`

### 规则层（可配置）

- 温度主规则：高温/温暖/微凉/偏冷四档
- 降雨叠加规则：雨天提升外套、防泼水相关品类权重
- 高湿叠加规则：抑制厚重针织，提升轻薄透气类
- 大风叠加规则：按季节提升外套/风衣/保暖类

### 输出层（双轨输出）

- 轨道 A：品类结构建议（面向备货）
  - 推荐品类列表
  - 品类汇总：总在仓、总在途、SKU 数、退货率均值、转化率均值
  - 采购建议：补货优先级、建议补货量、天气解释
- 轨道 B：推荐单品（面向执行）
  - 在库门槛过滤后，按推荐指数排序
  - 控制品类分散（每品类最多 N 个）
  - 输出款号、名称、库存、在途、推荐指数

### 评估层（后续可加）

- 规则命中覆盖率（命中规则后有货可卖的占比）
- 推荐品类售罄风险（低库存 SKU 占比）
- 推荐结果业务反馈（点击、加购、成交转化）
